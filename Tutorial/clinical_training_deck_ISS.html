What is an ISS?

An Integrated Summary of Safety (ISS) is a regulatory and clinical analysis framework that pools safety data across multiple studies to provide a broader view of a drug's adverse event profile.

  • It is not limited to one trial.
  • It usually includes several studies judged to be in scope for pooled safety analysis.
  • Common examples include pooling ADSL and ADAE across phase II/III or supportive studies.
  • The goal is a more stable, interpretable overall safety picture for clinicians, statisticians, and reviewers.

Slide 3

Why pool safety data across studies?

Clinical visibility

Rare or less frequent adverse events may be easier to observe when the denominator becomes larger across multiple studies.

Statistical stability

Pooling can improve precision of descriptive safety summaries and support more robust subgroup views.

Regulatory communication

Reviewers often want an integrated understanding of safety beyond any single protocol.

Important note: Pooling is not just stacking. Before datasets are appended, studies must be harmonized for variables, coding dictionaries, timing rules, treatment mapping, and analysis conventions.
Slide 4

Core datasets in a pooled ISS safety analysis

ADSL

  • One record per subject
  • Treatment assignment and pooled treatment variables
  • Dates, populations, and stratification context
  • Supports denominators and subject-level joins

ADAE

  • One or more records per subject adverse event
  • Terms, coded terms, seriousness, severity, timing
  • Often the main source for pooled safety summaries
  • Built from multiple study-level ADAE datasets in ISS
In this revised deck, the pooled ISS ADAE is explicitly shown as the result of combining ADAE from STUDY-001, STUDY-002, and STUDY-003 after standardization.

Slide 7

What must be harmonized before stacking?

Data standards

  • Variable names and labels
  • Variable types and lengths
  • Date and datetime formats
  • Controlled terminology and MedDRA version alignment

Analysis conventions

  • Treatment mapping to pooled groups
  • On-treatment timing rules
  • Population flags and inclusion criteria
  • Definitions for treatment-emergent adverse events
Practical teaching message: pooling = harmonize first, then append, then derive pooled analysis variables.
Slide 8

Programming workflow for pooled ISS ADAE

1

Identify in-scope studies

Select the studies appropriate for the integrated safety population and define inclusion rules.

2

Load each study's ADSL and ADAE

Bring in study-level subject and adverse event data separately for each protocol.

3

Standardize names, types, terminology

Align common columns and normalize controlled terminology and coding standards.

4

Derive common treatment and timing variables

Create pooled treatment variables and timing logic that work across studies.

5

Append / stack ADAE across studies

Concatenate standardized ADAE datasets into one multi-study pooled ADAE.

6

Merge subject context from pooled ADSL

Add subject-level treatment and population flags into the pooled AE structure.

7

Derive TRTEMFL and pooled groups

Apply integrated analysis definitions for treatment-emergent events and pooled categories.

8

Run QC and produce summaries

Validate the integrated dataset and generate tables, listings, and reviewer-ready outputs.

Slide 9

Illustrative SAS logic for multiple-study pooling

/* Study-level subject datasets */
data adsl_s1; set adam_study01.adsl; studyid="STUDY-001"; run;
data adsl_s2; set adam_study02.adsl; studyid="STUDY-002"; run;
data adsl_s3; set adam_study03.adsl; studyid="STUDY-003"; run;

/* Study-level AE datasets */
data adae_s1; set adam_study01.adae; studyid="STUDY-001"; run;
data adae_s2; set adam_study02.adae; studyid="STUDY-002"; run;
data adae_s3; set adam_study03.adae; studyid="STUDY-003"; run;

/* Standardize common variables for pooling */
data adsl_s1_std adsl_s2_std adsl_s3_std;
  set adsl_s1(in=a) adsl_s2(in=b) adsl_s3(in=c);
  length trt_pool $20;
  if trt01a in ("Drug A","Active","Treatment A") then trt_pool="Drug A";
  else if trt01a in ("Placebo","PBO") then trt_pool="Placebo";
  else trt_pool="Comparator";
  if a then output adsl_s1_std;
  else if b then output adsl_s2_std;
  else if c then output adsl_s3_std;
run;

data adae_s1_std adae_s2_std adae_s3_std;
  set adae_s1(in=a) adae_s2(in=b) adae_s3(in=c);
  length aedecod $200 aeterm $200;
  format astdt yymmdd10.;
  if a then output adae_s1_std;
  else if b then output adae_s2_std;
  else if c then output adae_s3_std;
run;

/* Append / stack multiple studies into pooled datasets */
data adsl_pool;
  set adsl_s1_std adsl_s2_std adsl_s3_std;
run;

data adae_pool;
  set adae_s1_std adae_s2_std adae_s3_std;
run;

proc sort data=adsl_pool; by studyid usubjid; run;
proc sort data=adae_pool; by studyid usubjid; run;

/* Merge pooled subject context into pooled ADAE */
data adae_iss;
  merge adae_pool(in=ae) adsl_pool(in=sl keep=studyid usubjid trt_pool trtsdt trtedt saffl);
  by studyid usubjid;
  if ae and saffl="Y";

  if astdt >= trtsdt then trtemfl="Y";
  else trtemfl="N";
run;
This is generic teaching code only. The central point is the explicit use of multiple study-level ADAE inputs and set adae_s1 adae_s2 adae_s3; for pooled construction.
Slide 10

Illustrative R logic for multiple-study pooling

# Study-level datasets already read into memory:
# study1_adsl, study2_adsl, study3_adsl
# study1_adae, study2_adae, study3_adae

standardize_adsl <- function(df, studyid_value) {
  df$STUDYID <- studyid_value
  df$TRT_POOL <- ifelse(df$TRT01A %in% c("Drug A","Active","Treatment A"), "Drug A",
                 ifelse(df$TRT01A %in% c("Placebo","PBO"), "Placebo", "Comparator"))
  keep <- c("STUDYID","USUBJID","TRT01A","TRT_POOL","TRTSDT","TRTEDT","SAFFL")
  df[, intersect(keep, names(df)), drop = FALSE]
}

standardize_adae <- function(df, studyid_value) {
  df$STUDYID <- studyid_value
  keep <- c("STUDYID","USUBJID","AETERM","AEDECOD","ASTDT","AESER")
  df[, intersect(keep, names(df)), drop = FALSE]
}

adsl_s1 <- standardize_adsl(study1_adsl, "STUDY-001")
adsl_s2 <- standardize_adsl(study2_adsl, "STUDY-002")
adsl_s3 <- standardize_adsl(study3_adsl, "STUDY-003")

adae_s1 <- standardize_adae(study1_adae, "STUDY-001")
adae_s2 <- standardize_adae(study2_adae, "STUDY-002")
adae_s3 <- standardize_adae(study3_adae, "STUDY-003")

# Append / stack multiple studies
adsl_pool <- rbind(adsl_s1, adsl_s2, adsl_s3)
adae_pool <- rbind(adae_s1, adae_s2, adae_s3)

# Merge pooled subject context into pooled ADAE
adae_iss <- merge(
  adae_pool,
  adsl_pool[, c("STUDYID","USUBJID","TRT_POOL","TRTSDT","TRTEDT","SAFFL")],
  by = c("STUDYID","USUBJID"),
  all.x = TRUE
)

adae_iss <- subset(adae_iss, SAFFL == "Y")
adae_iss$TRTEMFL <- ifelse(adae_iss$ASTDT >= adae_iss$TRTSDT, "Y", "N")
Equivalent logic can also be written using bind_rows() in a tidyverse workflow. The educational point is explicit multi-study selection, standardization, and row-binding.

From pooled ADAE to pooled safety tables

Typical summaries generated

  • Any treatment-emergent adverse event
  • Serious adverse events
  • Severe adverse events
  • AEs leading to discontinuation
  • System organ class / preferred term summaries

Clinical interpretation reminder

  • Counts are subject-based for many summary tables
  • One subject may have multiple event records
  • Pooled denominators must be defined consistently
  • Integrated timing rules affect treatment-emergent classification

Slide 13

Clinical reading of pooled safety outputs

Look at incidence

Start with how many subjects experienced at least one event in each pooled treatment group.

Look at severity and seriousness

Focus on medically important events, serious events, and severe events when discussing benefit-risk.

Look at comparability

Consider pooled treatment mapping and study composition when interpreting differences across groups.

Clinicians should remember that the pooled dataset reflects multiple contributing studies; the integrated result is broader than any single protocol.

Slide 15

Key take-home messages

  • Yes, ISS explicitly involves multiple-study handling.
  • Study-level ADAE datasets are not simply dumped together without preparation.
  • They are first harmonized for variables, coding, timing rules, and treatment mapping.
  • Then they are stacked vertically into one pooled ADAE structure.
  • Pooled ADSL provides subject-level context used to derive integrated safety analysis variables.
  • The final output supports clinician-facing and regulator-facing safety interpretation.
